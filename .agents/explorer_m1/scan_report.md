# تقرير الفحص الشامل لمستودع Plan+Note

تم إجراء فحص شامل لكافة ملفات التطبيق البرمجية في المسارات التالية:
- `app/`
- `components/`
- `contexts/`
- `lib/`

ويلخص هذا التقرير الثغرات، ومشاكل الأداء، وتنسيق الأكواد، ومقترحات التحسين في أربعة أقسام رئيسية.

---

## 1. جودة الكود والأخطاء البرمجية (Code Quality & Bugs)

### أ. حماية المسارات على جانب العميل فقط (Client-Side Route Guarding)
* **الموقع:** `app/dashboard/page.js`, `app/note/[id]/page.js`, `app/canvas/[id]/page.js`
* **الملاحظة:** يتم التحقق من حالة المستخدم والمصادقة في الـ client-side باستخدام `useEffect`:
  ```javascript
  useEffect(() => {
    if (!loading && !user) router.push('/');
  }, [user, loading, router]);
  ```
  هذا الأسلوب يتسبب في تحميل الصفحة وعرض المكونات لفترة وجيزة (فلاش أو وميض شاشة بيضاء/مؤشر تحميل) قبل التوجيه لصفحة البداية.
* **التوصية:** استخدام Next.js Middleware لحماية المسارات `/dashboard` و `/note` و `/canvas` لمنع تحميل الصفحة أو الكود الخاص بها بالكامل للمستخدمين غير المصرح لهم.

### ب. فقدان البيانات غير المحفوظة عند الانتقال الداخلي (Unsaved Changes Loss on Internal Routing)
* **الموقع:** `app/note/[id]/page.js` و `app/canvas/[id]/page.js`
* **الملاحظة:** يحتوي التطبيق على مستمع للحدث `beforeunload` لمنع فقدان البيانات عند إغلاق التبويب أو المتصفح. ولكن هذا المستمع **لا يُطلق** عند التنقل الداخلي في Next.js (مثال: الضغط على زر "رجوع" الذي ينفذ `router.push('/dashboard')`).
* **الخطورة:** في حال قام المستخدم بالكتابة ثم ضغط على "رجوع" فوراً قبل مرور وقت التخزين التلقائي (1.5 إلى 2 ثانية)، سيتم الانتقال وتفقد التعديلات الأخيرة تماماً دون أي تحذير.
* **التوصية:** تعديل زر "رجوع" ليتحقق أولاً من قيمة `hasUnsavedChanges`. إذا كانت `true` يتم إما إجبار الحفظ الفوري أولاً ثم الانتقال، أو عرض تحذير للمستخدم لتأكيد الخروج.

### ج. خطر انتهاء التوقيت بعد إلغاء تركيب المكون (State Updates on Unmounted Components)
* **الموقع:** `app/note/[id]/page.js` (السطر 73) و `app/canvas/[id]/page.js` (السطر 156)
* **الملاحظة:** في دالة الحفظ، يتم استدعاء `setTimeout` لتغيير حالة `saved` إلى `false` بعد ثانيتين. في حال انتقل المستخدم خارج الصفحة قبل انتهاء الثانيتين، سيتم محاولة تحديث الحالة لمكون غير موجود (Unmounted). على الرغم من أن React 18+ لا يعرض تحذيراً صريحاً بذلك، إلا أنه يمثل ممارسة غير جيدة في إدارة الذاكرة.
* **التوصية:** تخزين معرف المؤقت في `useRef` وإلغاؤه (clearTimeout) في دالة تنظيف الـ `useEffect` (Cleanup function).

## 2. الأداء وسيو الموقع (Performance & SEO)

### أ. غياب التوليد الاستاتيكي لصفحات المدونة (Lack of Static Generation for Blog Pages)
* **الموقع:** `app/blog/[slug]/page.js`
* **الملاحظة:** يتم قراءة المقالات بشكل ديناميكي من ملفات Markdown وقت الطلب (On-demand SSR) بدون استخدام `generateStaticParams`.
* **الأثر:** يؤدي هذا لزيادة زمن الاستجابة (TTFB) للمقالات وإضعاف أداء الـ SEO والـ LCP.
* **التوصية:** استخدام دالة `generateStaticParams()` لتوليد كافة مقالات المدونة استاتيكياً أثناء بناء التطبيق (Build Time)، حيث أن المقالات ثابتة وتُعرف مساراتها مسبقاً.

### ب. قراءة وتفكيك البيانات في كل إعادة صياغة (Initial Data Parsing on Every Render)
* **الموقع:** `app/canvas/[id]/page.js` (السطر 122 و 255)
* **الملاحظة:** يتم استدعاء الدالة `initialData={getInitialData()}` مباشرة داخل الـ JSX. تقوم هذه الدالة بعمل `JSON.parse(docData.content)` في كل دورة رندر للمكون.
* **الأثر:** بما أن لوحة Excalidraw تطلق حدث `onChange` بكثافة عالية (عند كل حركة ماوس، زوم، أو تحريك لوحة)، فإن الصفحة تعيد الرندر باستمرار، مما يعني استدعاء `JSON.parse` لبيانات اللوحة الكبيرة (التي قد تحتوي على مئات العناصر والصور) عشرات المرات في الثانية. هذا يتسبب في بطء شديد في الأداء (UI Lag).
* **التوصية:** تفكيك البيانات وتجهيزها **مرة واحدة فقط** عند جلب المستند، وحفظ النتيجة في `state` أو استخدام `useMemo` لمنع تكرار العملية.

### ج. حفظ تلقائي زائد عند الزوم والتحريك (Unnecessary Auto-Save on Zoom/Pan)
* **الموقع:** `app/canvas/[id]/page.js` (السطر 159)
* **الملاحظة:** يتم تشغيل `handleChange` عند أي حدث من Excalidraw بما في ذلك عمليات الـ Zoom والـ Panning. هذا يؤدي إلى وسم اللوحة بأنها غير محفوظة `setHasUnsavedChanges(true)` وحفظها سحابياً بعد ثانيتين حتى لو لم يقم المستخدم بإضافة أو تعديل أي عنصر!
* **الأثر:** استهلاك غير مبرر لعمليات الكتابة في Firebase (Firestore Write Limits) واستهلاك موارد الشبكة والمعالج للمستخدم.
* **التوصية:** مقارنة العناصر الحالية بالعناصر التي تم حفظها سابقاً، أو تصفية أحداث التغيير للتحقق من وجود تعديل فعلي في العناصر أو الصور قبل وسم الملف بأنه غير محفوظ.

### د. ضغط الصور المتكرر والمكثف (Repeated Base64 Image Compression on Save)
* **الموقع:** `app/canvas/[id]/page.js` (السطر 34 و 144)
* **الملاحظة:** يتم تمرير كافة ملفات اللوحة إلى دالة `compressFiles` عند كل عملية حفظ تلقائي. تقوم الدالة بالدوران حول الصور وفحص حجمها، وإذا كانت أكبر من 400KB تقوم بتحويلها إلى Blob وضغطها ثم تحويلها لـ Base64 مجدداً.
* **الأثر:** استهلاك معالج هائل، وبطء في عملية الحفظ التلقائي خاصة مع وجود صور متعددة.
* **التوصية:** ضغط الصور **مرة واحدة فقط** عند إضافتها للوحة (مثال: عن طريق ربطها بحدث الإضافة)، بدلاً من معالجة كافة الصور باستمرار عند كل عملية حفظ.

## 3. واجهة المستخدم وتجربة المستخدم (UI/UX)

### أ. تضارب في تكوين خط Cairo (Tailwind Font Family Discrepancy)
* **الموقع:** `tailwind.config.js` و `app/layout.js`
* **الملاحظة:** في `app/layout.js` يتم تحميل الخط عبر `next/font/google` وتعريفه كمتغير CSS باسم `--font-cairo`. ولكن في ملف `tailwind.config.js` تم تعريف الخط كالتالي:
  ```javascript
  fontFamily: {
    arabic: ['Cairo', 'sans-serif'],
  }
  ```
  هذا يعني أنه إذا تم استخدام الكلاس `font-arabic` في أي مكان بالتطبيق خارج المكون الرئيسي، سيبحث المتصفح عن الخط المثبت محلياً بالجهاز باسم `Cairo` ولن يستخدم النسخة المحملة عبر Next.js.
* **التوصية:** تعديل ملف `tailwind.config.js` ليشير إلى المتغير كالتالي:
  ```javascript
  fontFamily: {
    arabic: ['var(--font-cairo)', 'sans-serif'],
  }
  ```

### ب. تباين الألوان ومتطلبات إمكانية الوصول (Accessibility Contrast)
* **الموقع:** `app/dashboard/page.js` (السطر 197)
* **الملاحظة:** يُعرض نص تاريخ التعديل "آخر تعديل: ..." باستخدام الكلاس `text-gray-300` على خلفية بيضاء.
* **الأثر:** نسبة تباين الألوان هنا ضعيفة جداً وتخالف معايير WCAG AA لإمكانية الوصول (مما يجعل قراءتها صعبة لضعاف البصر).
* **التوصية:** استخدام كلاس تباين أفضل مثل `text-gray-500` أو `text-gray-400`.

## 4. الحماية والأمان (Security)

### أ. غياب ملف قواعد الأمان في المستودع (Missing Security Rules File)
* **الملاحظة:** لا يوجد ملف `firestore.rules` في الكود المصدري. هذا يعني أن قواعد الحماية تُدار حصراً من لوحة تحكم Firebase، مما يحرم المشروع من ميزة تتبع التغييرات (Version Control) لقواعد الأمان ومراجعتها.
* **التوصية:** تصدير القواعد وحفظها في ملف `firestore.rules` في جذر المشروع لتتبعها وإدراجها في خط النشر.

### ب. التحقق من صلاحيات الملفات في جانب العميل فقط (Client-Side Ownership Verification)
* **الموقع:** `app/note/[id]/page.js` (السطر 58) و `app/canvas/[id]/page.js` (السطر 114)
* **الملاحظة:** يتم جلب المستند كاملاً من Firestore باستخدام المعرف `id` أولاً، ثم يتم التحقق من مطابقة `userId === user.uid` على جانب العميل لإقرار الانتقال أو إعادة التوجيه.
* **الخطورة:** إذا كانت قواعد Firestore في السحابة غير مضبوطة بشكل صحيح (أو تم تعديلها بالخطأ لتسمح بالقراءة العامة)، يمكن لأي مستخدم جلب أي مستند آخر بمجرد معرفة الـ ID الخاص به دون أن تمنعه قاعدة البيانات.
* **التوصية:** التأكد من كتابة قواعد أمان Firestore مشددة تمنع القراءة تماماً للمستندات التي لا تطابق معرف المستخدم:
  ```javascript
  match /documents/{docId} {
    allow read, write: if request.auth != null && resource.data.userId == request.auth.uid;
  }
  ```

## الخلاصة والتوصيات القابلة للتنفيذ

1. **إصلاح عاجل للأداء:** تخزين ناتج `getInitialData()` في State عند التحميل الأول لمنع تفكيك الـ JSON بشكل متكرر أثناء التحريك والرسم.
2. **إصلاح تجربة المستخدم وسرعة الاستجابة للمدونة:** إضافة `generateStaticParams` لصفحات المقالات لتسريع فتحها وجعلها صديقة لمحركات البحث بالكامل.
3. **تحسين الحماية والتنقل:** تعديل أزرار الرجوع لحفظ التعديلات فوراً قبل الخروج، وضبط وتوثيق ملف قواعد Firestore لضمان حماية بيانات المستخدمين على مستوى الخادم لا العميل فحسب.
